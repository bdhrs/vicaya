## Retrospective
- [POSITIVE] Evidence: casting a wide net at Phase 2 (all 46 EBC-catalogue candidates, not just a pre-narrowed shortlist) via a single Haiku sub-agent meant Phase 5 synthesis had full Pāḷi+English+Āgama-pull evidence for every candidate before any selection was made, so the final ten were chosen from complete data rather than partial. Cause: dispatching the canon-search sub-agent with the whole candidate list instead of a top-N cut. Fix: keep doing this for future "identify N suttas" curation requests — the exclusion-checking and selection quality benefit directly.
- [WORKFLOW] Evidence: a targeted Pāḷi stem search ("subha" in the wrong book, s0203m_mul) returned MN135's alternate title "Cūḷakammavibhaṅgasuttaṃ subhasuttantipi vuccati" instead of MN99, and I nearly cited the wrong sutta's content under MN99's canon_ref. Cause: two suttas in the CST corpus share the interlocutor name "Subha" and one editions-tradition even doubles the title. Fix: `resolve-citation` caught it before drafting (Hard Rule 9 worked exactly as designed) — worth reinforcing in SKILL.md that duplicate-title collisions are a real, recurring failure mode, not just book-code mismatches.
- [CONFUSION] Evidence: none this run.
- [BEHAVIOR] Evidence: the library folders network share (filesrv1) was entirely unmounted for the whole run — 0 of ~160 library hits had `source_available: true`. I cited only the 2 sources whose exact bibliographic title was confirmable from the FTS snippet/filename itself (Van Gorkom, Mikkelson) and logged a third (Ornatowski) as unusable-for-formal-citation rather than guessing its full title from memory. Cause: environment issue, not a research shortcut. Fix: none needed in the skill; the existing `source_available` handling guidance was sufficient to make the right call without guidance changes.
- [BUG] Evidence: none this run — all helper calls behaved as documented.
- [DOC] Evidence: the cross-check reviewer (opencode/agy chain, not self-review) caught two commentary misattributions (Itivuttaka's aṭṭhakathā is Dhammapāla's Paramatthadīpanī, not Buddhaghosa's Manorathapūraṇī; SN1.42's is Sāratthappakāsinī not Manorathapūraṇī) that I had gotten wrong in the first draft by pattern-matching "the AN/Iti/SN commentary is whatever I called the last one." Cause: after citing Manorathapūraṇī correctly for AN sources, I re-used the same commentary name for adjacent Iti/SN sources without re-verifying which aṭṭhakathā each Nikāya/Khuddaka-text actually has. Fix: SKILL.md's book-code tables list `_att` codes per book but don't have a lookup table mapping book → commentary *name* (Papañcasūdanī/Manorathapūraṇī/Sāratthappakāsinī/Paramatthadīpanī/etc.) — a small reference table would have prevented this without needing the cross-check to catch it.

## Improvement suggestions
- Suggest: add a short reference table to SKILL.md (near the book-code map) mapping each Nikāya/Khuddaka-book to its aṭṭhakathā's actual name — DN/MN/SN/AN commentaries have different names (Sumaṅgalavilāsinī/Papañcasūdanī/Sāratthappakāsinī/Manorathapūraṇī) and several Khuddaka texts have their own named commentaries (Dhammapāla's Paramatthadīpanī covers Udāna, Itivuttaka, Vimānavatthu, Petavatthu, Theragāthā, Therīgāthā) — this is an easy misattribution to make by analogy from the last-cited commentary name, and a cross-check-only catch is later and more expensive than a lookup table.
